# Research Proposal: BEACON-GCL

## Problem Anchor

- **Bottom-line problem**：图对比学习中的 false-negative correction 不能只判断 pair 是否“像同类”；它还必须判断这个 pair edit 是否会破坏节点分类所需的边界几何。
- **Trigger evidence**：CAST Certificate Pilot-A 中，C4 latent certificate 低 overlap 于 kNN/PPR/CAST，但没有带来稳定 accuracy gain；C5 hybrid 还继承了较高 CAST overlap。
- **Must-solve bottleneck**：从 false-negative detection 转向 false-negative edit eligibility。
- **Non-goals**：不继续 CAST C4/C5 小修；不做普通 kNN/PPR/CAST mining；不做 loss-only reweighting；不把 smoke 写成性能 claim。

## Method Thesis

**One-sentence thesis**：BEACON-GCL accepts a false-negative edit only when a counterfactual boundary-utility gate predicts that neutralizing the pair reduces harmful repulsion without damaging local decision geometry, representation rank, or uniformity.

## Proposed Method

### 1. Candidate Pool Is Not the Contribution

Construct a broad but fixed candidate pool:

```text
P_i = topk_knn(i) union topk_ppr(i) union topk_cast(i) union topk_certificate(i)
```

This pool is deliberately not claimed as novel. It only supplies possible edits so the method can test eligibility under strong controls.

### 2. Counterfactual Edit Operator

For a candidate pair `(i, j)`, define a small virtual edit on frozen embeddings:

```text
z_i' = z_i + eta * (z_j - z_i)
z_j' = z_j + eta * (z_i - z_j)
```

or the corresponding neutralization action in the contrastive denominator. Smoke starts with the embedding edit because it is cheap and directly comparable to existing relation smoothing.

### 3. Utility Gate

BEACON computes:

```text
U(i,j) = Benefit(i,j)
       - lambda_b * BoundaryRisk(i,j)
       - lambda_g * GeometryRisk(i,j)
```

where:

- `Benefit(i,j)` estimates whether the current negative force conflicts with multi-view / feature / local consistency.
- `BoundaryRisk(i,j)` estimates whether the virtual edit reduces train-only or validation-calibrated probe margin; a fully unsupervised proxy uses local competitor separation.
- `GeometryRisk(i,j)` estimates rank loss, local uniformity loss, degree-bucket over-concentration, and accepted-pair quota pressure.

The action is:

```text
if U(i,j) > tau and BoundaryRisk(i,j) < rho:
    neutralize pair or allow weak positive smoothing
else:
    keep as ordinary negative
```

### 4. Why This Explains CAST Failure

CAST/C4 can find low-overlap pairs, but BEACON asks whether editing those pairs is safe for downstream geometry. If a low-overlap pair connects two different local boundary basins, then pulling or neutralizing it may reduce useful repulsion and hurt the Logistic Regression evaluator. BEACON treats such pairs as boundary-bearing negatives, not false negatives.

## Implementation Plan

### BEACON-lite Smoke

Use existing GRACE frozen embeddings and candidate matrices from `scripts/run_cast_certificate_smoke.py`.

Add a new runner rather than mutating CAST runner in-place:

```text
scripts/run_beacon_smoke.py
configs/beacon_smoke.yaml
```

The first runner may reuse helper functions from the CAST/IRIS runners:

- `create_or_load_split`
- `logreg_val_eval`
- `relation_smooth_embeddings`
- `topk_relations`
- `relation_overlap`
- existing kNN/PPR/CAST/C4 candidate construction

### First Systems

| ID | System | Purpose |
|---|---|---|
| B0 | GRACE | base reference |
| B1 | kNN smoothing | proximity control |
| B2 | CAST proxy smoothing | strongest previous mining control |
| B3 | C4 certificate smoothing | low-overlap certificate control |
| B4 | all-candidate neutralization | shows edit without eligibility |
| B5 | BEACON full gate | main method |
| B6 | shuffled utility gate | kills spurious utility |
| B7 | no-boundary gate | tests whether boundary risk matters |
| B8 | no-geometry gate | tests whether rank/uniformity risk matters |

## Diagnostics

- test@best-val under frozen Logistic Regression.
- accepted pair count per anchor.
- overlap with kNN/PPR/CAST/C4.
- label agreement, offline only.
- probe-margin delta on train/val only; no test labels in gate.
- representation rank delta.
- local uniformity delta.
- degree bucket accepted-pair concentration.
- utility distribution for accepted vs rejected pairs.

## Kill Rules

- B6 shuffled utility matches B5: kill BEACON.
- B7 no-boundary gate beats B5: boundary gate is unnecessary or harmful; revise.
- B4 all-candidate neutralization beats B5: eligibility gate is over-conservative; revise.
- B5 improves only by accepting pairs with high kNN/CAST overlap: collapse to mining; revise/pivot.
- B5 improves Cora seed0 but fails CiteSeer seed0 and PubMed seed0: do not enter Pilot-A.
- Any threshold chosen using test accuracy invalidates the run.

## Current Decision

`GO_TO_BEACON_SMOKE_PLANNING`

This is a mechanism-level pivot after CAST Pilot-A no-go. It supports only BEACON-lite smoke implementation, not Pilot-B, formal experiments, or paper claims.
